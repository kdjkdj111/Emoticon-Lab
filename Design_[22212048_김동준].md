## 1. Introduction

본 문서는 지난 Analysis 단계에서 정의한 요구사항을 기반으로, 사용자 경험(UX) 최적화와 시스템의 기술적 안정성을 동시에 구현하기 위한 'Emoticon-Lab'의 상세 설계 문서입니다.

본 시스템은 이모티콘 제작이라는 창작 과정을 기술적으로 지원하기 위해 고안되었으며, 본 시스템의 구조를 체계적으로 도식화하였습니다. 우선 클래스 다이어그램(Class Diagram)을 통해 시스템 아키텍처의 뼈대를 설계하였으며, 시퀀스 다이어그램(Sequence Diagram)으로 사용자 인터랙션과 기능적 흐름을 구체화하였습니다. 더불어, 상태 머신 다이어그램(State Machine Diagram)을 도입하여 복잡한 이미지 분석 프로세스 전반에서 발생하는 시스템의 상태 변화를 정의하였습니다.

또한, Emoticon-Lab은 사용자의 접근성을 고려하여 웹 서비스 기반의 아키텍처를 채택하였습니다. 본 명세서에 기술된 설계를 바탕으로 Emoticon-Lab은 단순한 검수 도구를 넘어, 예비 작가들에게 최적의 창작 환경을 제공하는 지능형 분석 플랫폼으로 발전할 것입니다. 본 문서는 시스템의 전반적인 구조를 상세히 제시하여, 시스템 구현을 위한 설계도가 됩니다.

## 2. Class diagram

본 프로젝트는 백엔드와 프론트엔드를 나누어 구현하는 구조로 설계되었습니다.
Spring Boot+ReactView 조합을 채택하였으며, 다이어그램 상단은 프론트엔드의 View class, 하단은 백엔드의 내부 연산 및 데이터 도메인을 담당합니다.
주요 View, Class 및 dataType을 명시하였고, 외부 서비스 및 연산은 일부 생략하였습니다.

<img src="resource/Design/Class/Full.png" width="100%" alt="Full Class Diagram">

각 클래스의 역할별로 패키지를 View, Controller, Service, Repository, Domain으로 구분하였습니다. View와 Controller간의 상호작용은 REST API 통신으로 연결됩니다.
Dto 패키지를 통하여 프론트 클래스와 백 클래스 간의 데이터 결합도를 낮추었습니다.

### 2.1 View class

본 다이어그램은 프론트엔드의 사용자 인터페이스를 구성하는 View class 구조를 나타냅니다. 본 프로젝트는 React 기반의 컴포넌트 구조를 전제로 설계되었습니다.

<img src="resource/Design/Class/View.png" width="100%" alt="View Layer Class Diagram">

전체적인 화면 흐름은 StartView(로그인/회원가입)에서 시작하여 프로젝트 목록을 볼 수 있는 DashboardView를 통해 UploadView로 가거나, WorkSpaceView로 바로 진입할 수 있는 구조입니다.
WorkSpaceView는 SimulatorView, TechnicalView, AIVIew를 포함하고 있으며 이는 activeTap 변수를 통한 탭 상태 전환 형태로 구성됩니다. 상위 뷰인 WorkSpaceView가 하위 세부 기능 뷰들과 생명주기를 공유하여 복합 관계를 가집니다.

또한 각 기능 뷰들은 구현에 필요한 PhoneSettings, ErrorSlot, AnalysisFormData 등 커스텀 dataType을 속성으로 가지며 처리됩니다.

## 3. Sequence diagram

본 챕터에서는 시퀀스 다이어그램을 활용하여 핵심 Use Case들을 사용할 때의 시스템의 흐름을 나타내는 냅니다.

### 3.1 Join

<img src="resource/Design/Sequence/UC1.png" width="100%" alt="UC1 Sequence Diagram">

1. 사용자 화면 진입 및 정보 입력
   - 사용자가 StartView에서 회원가입 버튼을 클릭하면, 회원가입 화면을 렌더링합니다.
   - 사용자가 닉네임, 이메일, 비밀번호를 입력한 뒤 가입하기 버튼을 누르면, handleSignup() 메서드가 실행되어 입력된 데이터를 취합합니다.

2. 인증 서비스 호출
   - StartView는 사용자로부터 입력받은 정보(email, password, nickname)를 인자로 담아, AuthService로 signUp 메서드를 호출합니다. 이를 통해 실제 데이터베이스에 계정 생성을 요청합니다.

3. 결과 분기 및 UI 처리 (alt 블록)
   - **가입 성공 시:**
     - AuthService가 계정 생성을 성공적으로 마치고, 생성된 User 객체를 반환합니다.
     - 화면에는 가입 성공 메시지가 출력되며, 사용자에게 즉시 로그인 화면을 렌더링합니다.
   - **가입 실패 시:**
     - 이메일 중복 등 특정 이유로 인해 AuthService가 error를 반환합니다.
     - 화면에 해당 에러 메시지를 표시하여 사용자가 입력한 정보를 다시 수정할 수 있도록 유도합니다.

---

### 3.2 Login

<img src="resource/Design/Sequence/UC2.png" width="100%" alt="UC2 Sequence Diagram">

1. 앱 시작 및 세션 확인
   - 사용자가 StartView에서 시작하기 버튼을 클릭하면, handleStartClick() 메서드가 실행됩니다.
   - StartView는 AuthService로 getSession() 메서드를 호출하여 세션을 확인합니다.

2. 세션 유무에 따른 분기 (첫 번째 alt 블록)
   - **세션 존재 시:**
     - DashboardView로 navigate(dashboard)를 호출하여 대시보드 화면을 렌더링합니다.
   - **세션 부재 시:**
     - StartView를 로그인 모드(mode = login)로 설정하고 로그인 화면을 렌더링합니다.

3. 로그인 요청 및 결과 분기 (두 번째 alt 블록)
   - 사용자가 이메일, 비밀번호를 입력 후 로그인 버튼을 누르면 handleLogin() 메서드가 실행됩니다.
   - StartView는 AuthService로 logIn(email, password) 메서드를 호출하여 인증을 요청합니다.
   - **로그인 성공 시 ([error == null]):**
     - DashboardView로 navigate(dashboard)를 호출하여 대시보드 화면을 렌더링합니다.
   - **로그인 실패 시 ([error != null]):**
     - 화면에 에러 메시지를 출력하여 재입력을 유도합니다.

### 3.3 Logout

<img src="resource/Design/Sequence/UC3.png" width="100%" alt="UC3 Sequence Diagram">

1. 로그아웃 요청
   - 사용자가 DashboardView에서 로그아웃 버튼을 클릭하면 handleLogout() 메서드가 실행됩니다.
   - DashboardView는 AuthService로 LogOut() 메서드를 호출하여 세션 종료를 요청합니다.

2. 세션 종료 및 화면 전환
   - AuthService가 세션 종료 처리를 완료하고 DashboardView로 결과를 반환합니다.
   - DashboardView는 StartView로 navigate(StartView)를 호출합니다.
   - 사용자에게 초기 화면(mode=landing)을 렌더링합니다.

### 3.4 Upload Image Set

<img src="resource/Design/Sequence/UC4.png" width="100%" alt="UC4 Sequence Diagram">

1. 이미지 파일 등록
   - 사용자가 UploadView에 이미지를 드롭하거나 파일을 선택하면 handleDrop(files) 메서드가 실행됩니다.
   - 화면에 이미지 프리뷰와 업로드된 파일 개수(uploadedCount)를 표시합니다.

2. 서버 저장 요청
   - 사용자가 '서버에 저장 후 분석 시작' 버튼을 클릭하면 uploadToServer() 및 handleStartInspection() 메서드가 실행됩니다.
   - UploadView는 WorkspaceController로 createProject()를 호출하여 프로젝트 생성을 요청합니다.

3. 저장 결과 분기 (alt 블록)
   - **업로드 성공 시:**
     - WorkspaceService는 ProjectRepository를 통해 프로젝트를 저장(save)합니다.
     - 전달받은 이미지 개수만큼 반복(loop)하여 EmoticonImageRepository에 개별 이미지를 저장합니다.
     - 모든 저장이 완료되면 View는 navigate(workspace, projectId)를 호출하여 워크스페이스 화면을 표시합니다.
   - **업로드 실패 시:**
     - 화면에 업로드 오류 메시지를 출력합니다.

### 3.5 Re-upload Modified Image

<img src="resource/Design/Sequence/UC5.png" width="100%" alt="UC5 Sequence Diagram">

1. 이미지 교체 요청 및 파일 선택
   - 사용자가 TechnicalView에서 '현재이미지 개별 교체' 버튼을 클릭하면 handleReuploadClick() 메서드가 실행됩니다.
   - 파일 선택기를 통해 사용자가 수정한 새 이미지를 선택하면 handleFileChange(File) 메서드가 실행됩니다.

2. 파일 검증 및 결과 분기 (alt 블록)
   - **파일 정상 시:**
     - TechnicalView는 WorkspaceView로 onUpdateImage(slotId, file)를 호출하여 개별 이미지를 업데이트합니다.
     - 내부적으로 uploadedImages 상태와 dynamicResults를 차례로 갱신합니다.
     - onSaveReport()를 실행한 뒤, 에러 리스트를 갱신하고 사용자에게 성공 알림을 표시합니다.
   - **파일 규격 불일치 시:**
     - 화면에 교체 실패 메시지를 출력합니다.

### 3.6 Request Technical Analysis

<img src="resource/Design/Sequence/UC6.png" width="100%" alt="UC6 Sequence Diagram">

1. 분석 시작 요청 및 상태 변경
   - 사용자가 TechnicalView에서 '분석 시작' 버튼을 클릭하면 handleStartAnalysis()가 실행됩니다.
   - 내부 상태를 분석 중(analysisStatus = analyzing)으로 변경하고 AnalysisController로 requestTechnicalAnalysis(projectId)를 호출합니다.

2. 데이터베이스 조회 및 이미지 검수 (loop 블록)
   - Controller는 TechnicalAnalysisService로 analyzeImages(projectId)를 호출합니다.
   - Service는 Repository를 통해 Project와 EmoticonImage 목록을 차례대로 조회합니다.
   - 조회된 각 이미지에 대해 반복문(loop)을 돌며 규격(checkDimensions), 여백(checkMarginViolation), 픽셀 찌꺼기(checkStrayPixels)를 검사하고 그 결과를 DB에 저장(save)합니다.

3. 검수 결과 반환 및 화면 분기 (alt 블록)
   - 검수가 완료된 TechnicalReport 객체가 View까지 반환됩니다.
   - **정상 완료 시:**
     - 상태를 완료(completed)로 변경하고 결과값(setDynamicResults)을 업데이트한 뒤 기술 리포트 화면을 렌더링합니다.
   - **서버 오류 시:**
     - 상태를 대기(ready)로 되돌리고 분석 실패 메시지를 출력합니다.
   - **네트워크 오류 시:**
     - 상태를 대기(ready)로 되돌리고 네트워크 오류 메시지를 출력합니다.

### 3.7 Request AI Analysis

<img src="resource/Design/Sequence/UC7.png" width="100%" alt="UC7 Sequence Diagram">

1. 분석 시작 요청 및 데이터 조회
   - 사용자가 AIView에서 폼을 입력하고 분석 시작 버튼을 클릭하면 requestAnalysis()가 실행됩니다.
   - 내부 상태를 분석 중(aiStatus = analyzing)으로 변경하고 AnalysisController로 requestAIAnalysis()를 호출합니다.
   - Controller는 AIAnalysisService의 analyzeWithGemini()를 호출하고, Service는 Repository를 통해 Project와 이미지 목록을 차례로 조회합니다.

2. 프롬프트 구성 및 외부 통신
   - AIAnalysisService는 조회된 데이터를 바탕으로 내부에서 프롬프트와 페이로드를 구성합니다.
   - ExternalAI 시스템으로 requestAnalysis(payload)를 호출하여 외부 API 서버와 통신합니다.

3. API 응답 결과 분기 (alt 블록)
   - **Gemini API 응답 성공 시 ([Gemini API 응답 성공]):**
     - 반환된 JSON 데이터를 파싱하여 AIReportRepository에 저장(save)합니다.
     - UserRepository를 호출하여 사용자의 API 호출 횟수를 증가시키고 DB에 갱신합니다.
     - 최종 AIReport가 View로 반환되면, 상태를 완료(completed)로 변경하고 데이터를 세팅한 뒤 AI 리포트 화면을 렌더링합니다.
   - **API 호출 실패 시:**
     - Timeout Error가 반환되면 상태를 대기(ready)로 되돌리고 호출 실패 오류 메시지를 출력합니다.
   - **API 할당량 초과 시:**
     - Too Many Requests 에러가 반환되면 상태를 대기(ready)로 되돌리고 할당량 초과 오류 메시지를 출력합니다.

## 4. State Machine Diagram
Client side(Front)와 System side(Back)두 가지로 나누어 설명하겠습니다.

### 4.1 Client Application State Machine Diagram

<img src="resource/Design/State Machine/Client.png" width="100%" alt="Client State Machine Diagram">

**1. 초기 인증 및 메인 대시보드 흐름**
- **StartView:** 시스템의 최초 진입점으로 로그인 및 회원가입 UI를 렌더링합니다. 회원가입 성공 시 로그인 탭으로 전환되며, 로그인 성공 시 DashboardView로 이동합니다.
- **DashboardView:** 진입 시 getUserProjects 메소드를 호출하여 프로젝트 목록을 로드합니다. 조회 실패 시 에러 알림 후 대시보드 상태를 유지합니다. 로그아웃 클릭 시 세션을 초기화하고 StartView로 복귀하며, '새 프로젝트 시작' 클릭 시 UploadView, 기존 프로젝트 클릭 시 WorkspaceView로 이동합니다.

**2. 이미지 업로드 흐름**
- **UploadView:** 진입 시 32개 슬롯 그리드를 렌더링합니다. 사용자가 이미지를 업로드 및 업로드 버튼 클릭 후 서버 저장에 성공하면 WorkspaceView로 이동하고, 실패 시 에러 알림과 함께 UploadView 상태를 유지합니다.

**3. 작업 공간 내부 복합 상태 흐름 (WorkspaceView)**
- **WorkspaceView:** 프로젝트 상세 조회를 수행하는 복합 상태입니다. 진입 후 데이터 로드에 실패하면 에러 알림 후 즉시 DashboardView로 복귀합니다. 내부에서 탭 간 이동 시 이전 작업 상태(로딩 상태 및 조회 데이터)를 초기화 없이 그대로 유지함을 히스토리 상태로 정의하였습니다. 
  - **SimulationView:** 워크스페이스 진입 시 최초 렌더링되는 상태로, 메신저 테마 설정을 로드하고 채팅 UI 시뮬레이션을 수행합니다.
  - **TechnicalView:** 기술 검수를 처리하는 복합 상태입니다. 초기 안내 상태(TechReady)에서 분석을 시작하면 로딩 상태(TechAnalyzing)로 진입하여 백엔드 응답을 대기합니다. 분석 성공 시 결과 화면인 TechCompleted로 전환되며, 통신 에러 발생 시 TechReady로 복귀하여 재시도를 유도합니다.
  - **AIView:** AI 분석을 처리하는 복합 상태입니다. 폼 입력 상태(AIReady)에서 분석을 요청하면 로딩 상태(AIAnalyzing)로 진입하여 외부 Gemini API의 응답을 대기합니다. 응답 수신 성공 시 AI 리포트 화면인 AICompleted로 전환되며, 할당량 초과 및 에러 발생 시 AIReady로 복귀합니다.

**4. 이탈 흐름**
- 사용자가 WorkspaceView에서 뒤로가기(handleSessionExit)를 수행하면 안전하게 DashboardView 상태로 복귀하며 전체 생명주기가 마무리됩니다.

### 4.2 Server System State Machine Diagram

<img src="resource/Design/State Machine/System.png" width="100%" alt="Server State Machine Diagram">

**1. 중앙 대기 및 라우팅 흐름**
- **Idle:** 서버 시스템은 상시 구동 상태로 API 요청이 들어올때까지 대기합니다. 클라이언트로부터 기능별 요청이 들어오면 이를 파싱하여 RequestProcessing 복합 상태로 진입합니다. 모든 비즈니스 로직의 정상 수행이나 예외 처리가 완료되면 항상 이 상태로 안전하게 복귀하여 다음 API 요청을 대기합니다.

**2. 요청 처리 복합 상태 흐름 (RequestProcessing)**
- **RequestProcessing:** 서버의 핵심 비즈니스 로직들을 하나로 캡슐화한 거대한 복합 상태입니다. 진입 시 라우터(Choice Node)를 통해 요청 엔드포인트에 따라 5개의 세부 상태로 분배됩니다. 각 작업이 성공적으로 마무리되어 종료 노드에 도달하면 밖으로 빠져나가 Idle 상태로 복귀하지만, 내부의 어느 상태에서든 에러가 발생하면 하던 작업을 즉시 중단하고 ErrorHandling 상태로 이탈합니다.
  - **Authenticating:** 로그인 요청 시 진입하며, 요청받은 메소드를 호출하여 DB 연산을 수행합니다. 인증 성공 시 세션을 발급하고 종료 노드로 이동합니다.
  - **ProjectManaging:** 프로젝트 CRUD 요청 시 진입하며, 파라미터를 파싱하고 해당 비즈니스 로직을 수행합니다. 수행 성공 시 요청 결과를 반환하며 종료 노드로 이동합니다.
  - **ImageUploading:** 이미지 업로드 요청 시 진입하며, 이미지 데이터들을 파싱하고 파일 스토리지에 개별 저장합니다. 업로드 성공 시 종료 노드로 이동합니다.
  - **TechnicalAnalyzing:** 기술 검수 요청 시 진입하며, DB에서 대상 이미지를 로드한 뒤 기술 분석 알고리즘을 수행하고 결과를 리포트로 저장합니다. 검수 성공 시 종료 노드로 이동합니다.

**3. 비동기 AI 분석 흐름**
- **AI Requesting:** AI 분석 요청 시 진입하며, 클라이언트가 보낸 FormData를 파싱하여 외부 API 전송 객체를 생성합니다.
- **AI_Waiting:** 외부 Gemini API로 분석 요청을 전송한 후, 타임아웃 타이머를 시작하고 백그라운드에서 API 응답을 대기(비동기)합니다.
- **AI_Saving:** 외부 API 응답을 성공적으로 수신하면 진입하며, JSON 응답을 파싱하여 AI Report를 생성하고 DB에 저장합니다. 분석 성공 시 종료 노드로 이동합니다.

**4. 예외 처리 및 이탈 흐름**
- **ErrorHandling:** RequestProcessing 내부에서 네트워크 장애, 타임아웃, 권한 없음, 데이터베이스 접근 에러 등의 예외가 발생할 경우 그룹 전이(Group Transition)를 통해 즉시 진입하는 에러 전담 상태입니다. 진입 시 에러를 Catch하고 원인을 분석하여 클라이언트에게 반환할 에러 메시지를 생성합니다. 예외 처리가 완료되면 적절한 상태 코드를 반환하며 다시 Idle 상태로 복귀하여 서버가 죽지 않고 생명주기를 이어가도록 마무리됩니다.

## 6. Glossary
- View (뷰 계층): 클라이언트의 사용자 인터페이스(UI)를 구성하는 React 기반의 컴포넌트 계층이다. 화면 렌더링 및 사용자 입력 이벤트 처리를 전담한다.

- Controller (컨트롤러 계층): 클라이언트의 REST API 요청을 수신하여 엔드포인트를 식별하고, 비즈니스 계층으로 라우팅한 뒤 연산 결과를 DTO 형태로 클라이언트에 응답하는 인터페이스 계층이다.

- Service (서비스 계층): 시스템의 핵심 비즈니스 로직과 분석 알고리즘을 수행하는 도메인 연산 계층이다. 트랜잭션 관리를 전담하며, 외부 API 통신 및 기술 검수 로직을 제어한다.

- Repository (리포지토리 계층): Spring Data JPA를 기반으로 데이터베이스에 접근하여 객체의 CRUD(생성, 조회, 수정, 삭제) 연산을 수행하는 데이터 액세스 계층이다.

- DTO (Data Transfer Object): 계층 간 데이터 교환 시 시스템 결합도를 낮추기 위해 사용하는 데이터 객체이다. 엔티티(Entity)를 화면에 직접 노출하지 않고, 필요한 속성만 정제하여 전송하는 역할을 한다.

- Entity / Domain (엔티티 도메인 객체): 데이터베이스 테이블과 1:1로 매핑되어, 고유 식별자(ID)와 영속적인 생명주기를 가지는 핵심 비즈니스 데이터 모델이다.

- Gemini API: Google의 대형 언어 모델(LLM) API. 본 시스템에서는 이모티콘의 컨셉, 시인성, 윤리적 리스크 등을 정성적으로 평가하는 외부 AI 분석 엔진으로 활용된다.

- Sequence Diagram (시퀀스 다이어그램): 특정 유스케이스 내에서 객체 간의 상호작용과 메시지 호출 흐름을 시간 순서에 따라 시각화한 동적 모델링 다이어그램이다.

- State Machine Diagram (상태 머신 다이어그램): 컴포넌트나 시스템이 생명주기 동안 외부 이벤트에 의해 겪는 상태 변화와 전이(Transition) 규칙을 규정한 UML 다이어그램이다.

- History State (히스토리 상태, H): 복합 상태(Composite State) 이탈 후 재진입 시, 초기화되지 않고 마지막으로 활성화되었던 하위 상태를 기억하여 복원하는 상태 머신 모델링 기법이다.
